Microsoft Azure Proof of Concept (PoC) Scope Document:
Legacy Web Application Migration and Modernization
Document Version: 1.0
Date: December 18, 2025
Prepared For: Microsoft Azure
Prepared By: Aetherius AI
1. Executive Summary
This document outlines the scope for a Proof of Concept (PoC) aimed at demonstrating the viability, benefits,
and practical considerations of migrating a legacy, on-premises web application and its associated database to
Microsoft Azure. Furthermore, the PoC will explore an initial step towards modernization by refactoring a key
component of the application to leverage Azure Platform-as-a-Service (PaaS) capabilities. The primary goal is
to validate the technical feasibility, assess performance, identify potential challenges, and provide a clear
understanding of the architectural patterns and operational models required for a full-scale cloud adoption.
This PoC will serve as a critical foundation for strategic decision-making regarding future cloud initiatives,
offering insights into scalability, cost optimization, security, and developer agility that Azure can provide.
2. Goals and Objectives
The overarching goal of this PoC is to de-risk a broader cloud migration strategy and gain hands-on experience
with Microsoft Azure services. Specific, measurable objectives include:
2.1 Technical Feasibility Validation: Successfully deploy the existing "Orders Management" web application
and its SQL database onto Azure infrastructure.
2.2 Performance Baseline Establishment: Measure and compare key performance indicators (KPIs) such as
application response times, database query speeds, and system throughput in the Azure environment against
the current on-premises benchmarks.
2.3 Cost Estimation Refinement: Gather data to generate more accurate cost projections for running the
migrated application in Azure, considering various service tiers and consumption models.
2.4 Security Posture Assessment: Implement baseline Azure security controls and validate their effectiveness
in protecting the migrated application and data.
2.5 Operational Familiarization: Provide key technical personnel with practical experience in deploying,
managing, and monitoring resources within the Azure ecosystem.
2.6 Modernization Viability Assessment: Successfully refactor the "Order Status Update" module into an Azure
Function, integrating it with the migrated application, to evaluate the benefits of serverless computing and
microservices architecture.
2.7 Disaster Recovery & High Availability Exploration: Demonstrate basic high availability principles for the
database component within Azure.
3. Scope – In Scope
The following components and activities are explicitly included within this PoC:
3.1 Application Component: The "Orders Management" web application (e.g., .NET Framework ASP.NET
application) will be the primary application targeted for migration.
3.2 Database Component: The existing SQL Server database (e.g., SQL Server 2016) supporting the "Orders
Management" application will be migrated.
3.3 Azure Services:
Compute: Azure Virtual Machines (IaaS) for the initial application server migration. Azure App Service (PaaS)
for the modernized component. Azure Functions (Serverless) for the modernized "Order Status Update"
module.
Database: Azure SQL Database (PaaS) or Azure SQL Managed Instance for the migrated database.
Networking: Azure Virtual Network (VNet), Network Security Groups (NSGs), and Private Endpoints for secure
connectivity.
Storage: Azure Blob Storage for application binaries and potential auxiliary data.Monitoring: Azure Monitor and Application Insights for performance, health, and logging.
Identity: Integration with Azure Active Directory for application authentication/authorization.
DevOps (Basic): Use of Azure DevOps Pipelines for automated deployment of the modernized component.
3.4 Migration Approach: "Lift-and-shift" for the initial web application and database, followed by a targeted
"replatforming/refactoring" for the "Order Status Update" module using Azure Functions.
3.5 Functionality Testing: Basic end-to-end user acceptance testing (UAT) for core application workflows within
the Azure environment.
3.6 Security Controls: Implementation of baseline Azure security best practices, including NSG rules, firewall
rules, identity management, and data encryption at rest and in transit.
3.7 Documentation: Creation of a PoC Summary Report detailing findings, lessons learned, performance
metrics, and recommendations.
4. Scope – Out of Scope
To maintain focus and manage the timeline, the following items are explicitly out of scope for this PoC:
4.1 Full-Scale Production Migration: This PoC is not intended to be a production-ready deployment or a
comprehensive migration of all organizational applications.
4.2 Comprehensive Application Modernization: Only the "Order Status Update" module is targeted for
modernization; the rest of the application will be migrated as-is.
4.3 Complex Integrations: No integrations with other external systems beyond the core application
components.
4.4 Advanced Azure Services: Advanced services like Azure Kubernetes Service (AKS), Azure Data Factory,
Azure Machine Learning, or complex IoT solutions are not included.
4.5 Advanced Disaster Recovery/Business Continuity: While basic high availability for the database will be
explored, a full multi-region disaster recovery solution is out of scope.
4.6 Performance Tuning beyond Basic Optimization: While performance will be monitored, extensive
performance tuning and optimization beyond initial configuration adjustments are excluded.
4.7 Data Archiving/Retention Policies: Detailed implementation of data archiving or complex retention policies
is not part of this PoC.
4.8 End-User Training: No formal end-user training on the Azure environment or new application interfaces.
4.9 Detailed Cost Optimization beyond Initial Assessment: While cost data will be gathered, a full-scale cost
optimization exercise is out of scope.
5. Use Cases / Scenarios
The PoC will focus on validating the following key scenarios:
5.1 Application Deployment: Successful deployment of the legacy "Orders Management" web application onto
an Azure Virtual Machine.
5.2 Database Migration & Connectivity: Migration of the SQL Server database to Azure SQL Database/Managed
Instance and establishing secure connectivity from the web application.
5.3 Core Application Functionality: Verification of critical business workflows within the migrated "Orders
Management" application (e.g., creating a new order, viewing order history, user login).
5.4 Modernized Module Integration: Successful deployment and integration of the "Order Status Update"
Azure Function with the core application.
5.5 Performance Monitoring: Real-time monitoring of application and database performance using Azure
Monitor and Application Insights.
5.6 Security Validation: Testing of basic access controls and network isolation to ensure data integrity and
confidentiality.
5.7 High Availability (Database): Demonstrate automatic failover for the Azure SQL Database in case of a
service disruption.
6. Success Criteria
The PoC will be deemed successful if the following criteria are met:6.1 Application Accessibility: The "Orders Management" web application is accessible and functional from
designated test client machines within Azure, with a target uptime of 99.9%.
6.2 Performance Equivalence/Improvement: Application response times and database query performance
within Azure are equivalent to or better than current on-premises benchmarks. (Specific metrics to be defined
at PoC commencement).
6.3 Data Integrity: All migrated data remains consistent and accurate post-migration.
6.4 Modernized Module Functionality: The "Order Status Update" Azure Function executes correctly and
integrates seamlessly with the main application, performing its intended logic.
6.5 Security Compliance: All defined baseline security controls are implemented and verified.
6.6 Operational Visibility: Azure Monitor and Application Insights successfully collect and display relevant
metrics, logs, and alerts for the application and database.
6.7 Cost Insight: A clear preliminary cost estimate for running the migrated application in Azure is generated.
6.8 Knowledge Transfer: Technical team members gain practical experience in deploying and managing Azure
resources relevant to the PoC.

13. Risk Assessment (High-Level)
13.1 Application Compatibility Issues: The legacy application may have unforeseen dependencies or
configurations not directly transferable to Azure.
Mitigation: Thorough pre-PoC assessment, starting with a non-critical application, and ensuring rollback
plans.
13.2 Performance Degradation: The application may not perform as expected in Azure without significant rearchitecture.
Mitigation: Careful selection of Azure VM/database tiers, thorough testing, and establishing performance
baselines.
13.3 Security Vulnerabilities: Misconfiguration of Azure security controls could expose data.
Mitigation: Adherence to Azure security best practices, regular security reviews, and least privilege access.
13.4 Budget Overruns: Unforeseen Azure consumption costs.
Mitigation: Setting up Azure budgets and alerts, monitoring consumption regularly, and right-sizing resources.
13.5 Resource Availability: Key personnel (client or PoC team) may not be consistently available.
Mitigation: Clear communication channels, dedicated time slots, and early identification of dependencies.
