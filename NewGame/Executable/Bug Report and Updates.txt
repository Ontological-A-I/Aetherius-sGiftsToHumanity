Bug Report (Fixed Issues)
CRITICAL CRASHES
Physics Engine Crash (ItemType.MONSTER):
Issue: The game crashed when calculating fall damage or attacks because ItemType.MONSTER did not exist.
Fix: Updated physics_validation_parser.py to use isinstance(entity, Monster) instead of checking an Enum that doesn't exist for entities.
Circular Import Crash:
Issue: combat_pattern_recognition.py tried to import entities.py, which imported other files, causing a loop that prevented the game from starting.
Fix: Removed the unnecessary BodyPartStatus import from the combat engine.
Save Loading Crash:
Issue: Loading a game often failed silently or crashed if the JSON was slightly off.
Fix: Added traceback error logging to world_manager.py so specific errors are printed to the console during load failures.
PERSISTENCE (SAVING/LOADING)
Item Amnesia (Torch Durability):
Issue: Torches and batteries reset to full charge/durability every time the game was loaded.
Fix: Updated entities.py (Item.to_dict) to explicitly save the props dictionary.
Monster Wolverine Healing:
Issue: Injured monsters (e.g., a zombie with a missing leg) would fully heal and regenerate limbs upon reloading the save.
Fix: Updated entities.py (Monster.to_dict) to serialize body part status.
Scent Trail Erasure:
Issue: Scent trails relied on real-world time (time.time()). If you saved and loaded an hour later, all scents vanished immediately.
Fix: Refactored scent_tracking_system.py to use Step-Based Decay (distance in list index) instead of Time-Based Decay.
Campfire Degradation:
Issue: Saved Campfires loaded back in as generic "Rubble" structures, losing their fuel and fire status.
Fix: Added polymorphism logic in Location.from_dict to detect and restore Campfire objects specifically.
GAMEPLAY & LOGIC
Attack Parsing Failure:
Issue: Typing attack rotting shambler head failed because the parser thought the monster's name was just "rotting".
Fix: Updated cmd_attack in game_engine.py to intelligently split the string, extracting known limb names from the end and using the rest as the name.
Container Looting:
Issue: The take command only looked at the floor, ignoring items inside open fridges or crates.
Fix: Updated cmd_take to iterate through all unlocked containers in the room.
Container Visibility:
Issue: The open command didn't actually list what was inside the container.
Fix: Updated cmd_open to print the inventory list of the target container.



Update Report (New Features)
1. Self-Installation System (main.py)
Feature: The game now behaves like a professional installer.
Behavior: When SurvivalRPG.exe is run, it checks if it is located in C:\Games\SurvivalRPG. If not, it automatically creates that directory, copies itself there, and launches the installed version.
2. Robust Save Directory Management (world_manager.py)
Feature: Forced centralization of save files.
Behavior: The game now forces save data to be written to C:\Games\SurvivalRPG\save_data.
Fallback: If the game lacks Administrator privileges to write to C:/, it automatically falls back to a local save_data folder next to the executable, ensuring no data is lost.
3. Enhanced Narrator
Feature: Visual durability cues.
Behavior: Items on the ground now display their state in the room description (e.g., Makeshift Torch [24 turns] instead of just Makeshift Torch), allowing players to decide if an item is worth picking up.
4. System Hooks
Feature: Preparation for future expansion.
Behavior: The FractalEngine and CombatTacticsEngine have been wired up with save() and load() methods that accept the centralized directory path, ensuring that when you expand these systems later, they will persist correctly without further code changes.
